# CLAUDE.md — LollaSchedule

Context + handoff for working on this repo across sessions. Read this first.

## What this is
A Next.js web app that connects to **Spotify**, analyzes the user's listening
history, and generates an **optimal Lollapalooza 2026 schedule** (Grant Park,
Chicago · July 30 – Aug 2, 2026) — scoring every artist on the lineup against the
user's taste and resolving stage conflicts with a travel-time-aware optimizer.

Stack: **Next.js 15 (App Router) + TypeScript**, Spotify-only (Apple Music is a
future add-on), no client secret (PKCE).

## Run it
```bash
npm install
npm run dev          # binds to 127.0.0.1 on purpose (see gotcha below)
```
Open **http://127.0.0.1:3000** → Connect Spotify → dashboard → "Build my schedule".

- `.env.local` (gitignored) holds `SPOTIFY_CLIENT_ID` + `SPOTIFY_REDIRECT_URI`.
  The Client ID is already set locally. App is registered at
  developer.spotify.com under this account.

## Deployment (LIVE: https://lollaschedule.vercel.app)
Deployed on Vercel via GitHub auto-deploy (push to `main` → deploy). It works
in prod AND local because OAuth is now **host/protocol-aware**: `redirectUri(req)`
/ `originFromRequest(req)` in lib/spotify.ts derive the origin from the request
(`host` + `x-forwarded-proto`), and cookies set `secure` only on https. The PKCE
redirect_uri must match exactly across /login and /callback — both derive it the
same way, so don't reintroduce a hardcoded URL.
- **Spotify dashboard** must list BOTH redirect URIs: `http://127.0.0.1:3000/callback`
  (dev) and `https://lollaschedule.vercel.app/callback` (prod).
- **Vercel env**: `SPOTIFY_CLIENT_ID` must be set (Production scope) — a missing
  var was the cause of a prod 500; /login now fails readably to
  `/?error=missing_client_id` instead. Redeploy after changing env vars.
- `SPOTIFY_REDIRECT_URI` is NOT needed in prod (derived from request).
- enriched-cache.json is committed (warm), so reads work on Vercel's read-only
  FS; cache writes fail silently (best-effort) — fine.

## Architecture / where things live
```
app/
  page.tsx              landing + Connect button (+ localhost→127 client guard in layout)
  login/route.ts        PKCE: generates verifier+challenge, redirects to Spotify
  callback/route.ts     exchanges code → tokens, sets httpOnly cookies
  dashboard/page.tsx    shows top 50 artists (the raw taste signal)
  schedule/page.tsx     server: scores lineup, hands serializable payload to client
  schedule/ScheduleClient.tsx  client: timeline (stages=columns), click-to-lock overrides,
                               re-optimizes in-browser, .ics/share/print export
  share/page.tsx        public read-only STATIC snapshot from ?s=<bitset> (or legacy
                        ?sets=csv) — no auth, print-friendly, noindex
  share/[slug]/page.tsx public read-only LIVE link — re-reads KV every request, so it
                        follows the owner's edits; no-store, noindex
  share/ShareView.tsx   the schedule renderer both share routes share
  api/share/save        publish current picks to your live link (creates on first use)
  api/share/revoke      turn the live link off everywhere it was pasted
  api/share/resolve     batch slug → current picks (friends panel refresh)
lib/
  spotify.ts            PKCE helpers, token exchange, API client (top artists/tracks, search)
  taste.ts              buildTasteProfile(token) → affinityByName + genreWeights
  enrich.ts             enrichArtists() → Spotify search for genres/images, disk-cached
  scoring.ts            scoreArtist() → score 0..100 + tier (must-see/worth-it/discovery/skip)
  optimizer.ts          optimizeDay(sets, lockedIds?) → weighted interval scheduling +
                        travel penalty; locked ids are force-included (overrides)
  ics.ts                buildIcs() → .ics feed; converts local CDT (UTC-5) to UTC Z-times
  lineup.ts             loads data/lineup-2026.json
  setcode.ts            stateless share codec (bitset ?s=) + parseShareSlug for live links
  sharelink.ts          LIVE share links: slug mint, KV record, server-derived ownership
data/
  lineup-2026.json      REAL 2026 schedule: 172 sets, 7 stages (generated by build-lineup.mjs)
  build-lineup.mjs      source-of-truth set list + parser; re-run to regenerate the JSON
  stage-distances.json  walking-minutes positions per stage (tunable estimate)
  enriched-cache.json   (generated at runtime) cached Spotify genre lookups
```

## Scoring model (lib/scoring.ts + lib/taste.ts)
`score = max(70*directAffinity + 30*genreMatch, discoveryPotential) + 8*emphasis`.
- **directAffinity** (0..1, window-INDEPENDENT) drives tier/color. Built in
  taste.ts from the signals below; a favorite stays a favorite in every window.
- **emphasis** (selected window only) is a small ranking nudge (≤8 pts), never
  changes tier/color.
- **genreMatch is effectively DEAD** — see gotcha below; it's ~always 0.
- Tiers: 🔥 must-see (direct ≥ .5), 🔵 worth-it (≥ .15), 🟣 discovery, 🟠 wildcard.

### Affinity from signals (taste.ts buildTasteProfile) — current weights
| Signal | Affinity | Tier it lands |
|---|---|---|
| Top artist rank 1 → 50 | 1.0 → ~0.02 | must-see→down |
| Top track (artist credited) | up to 0.6 | varies |
| Recently played (last 50) | +0.12/hit, cap 0.4, additive | boosts |
| **Liked song** (saved, up to 5000) | floor 0.2 (1 like) → 0.45 (many) | worth-it+ |
| **Following** | floor 0.2 | worth-it |
- KEY RULE the user set: **1 liked song == following == 0.2 (worth-it)**. A
  follow is a LIGHT signal — must NOT be must-see (that bug shipped once: all
  84 follows showed green; fixed by dropping the floor 0.5→0.2).
- `taste.sourcesByName` (Map<name, Set<signal>>) records provenance for debug.

## Optimizer (lib/optimizer.ts)
Classic weighted-interval-scheduling DP per day, sorted by end time, with a
`transitionPenalty` that subtracts value when you can't walk between stages
before the next set starts (uses `data/stage-distances.json`).

## Gotchas (important)
- **`localhost` vs `127.0.0.1`:** Spotify bans `localhost` in redirect URIs, so
  the whole app MUST run on `127.0.0.1`. The registered redirect URI is
  `http://127.0.0.1:3000/callback`. We pin this three ways: dev server binds to
  127.0.0.1, a client-side guard in `app/layout.tsx` bounces localhost→127, and
  callback redirects are built from the `Host` header (Next's `req.url` lies and
  says `localhost` in dev). Don't "simplify" these away.
- **Spotify Development Mode:** the app only lets allowlisted accounts log in (25
  max). Add users at dashboard → app → Settings → User Management. Symptom if
  missing: "user not registered" on login.
- **Never run `npm run build` while `npm run dev` is running.** They share the
  `.next` directory, so the production build swaps out the chunks the running dev
  server has loaded. The browser then shows `Runtime TypeError: Cannot read
  properties of undefined (reading 'call')` and the server log says `Could not
  find the module ... in the React Client Manifest` — on *every* page, including
  ones you never touched, which makes it look like a code bug. Cure: stop dev,
  `rm -rf .next`, restart. (Hit this on 2026-07-19.)
- Middleware was tried for the localhost redirect and removed — Next normalizes
  the Location header back to a relative path, so it can't move hosts. Use the
  client-side guard instead.
- **Spotify genres + popularity are GONE for this app (confirmed 2026-06-22):**
  `/search` returns empty genres + popularity 0, `/artists?ids=` returns **403**
  in dev mode, AND `/me/top/artists` now also returns **0 genres** (probe logged
  `0 distinct genres; 0 top artists have genres`). So genre-based discovery is
  impossible — the ONLY working signals are direct listening (top/recent/saved/
  followed). This is the whole reason the AI predictor (next step) exists.
- **Debugging taste:** set `WATCH_ARTIST=<substr>` env (default "claire") — the
  taste build logs whether that artist appears in ANY raw signal + exact
  spelling, plus per-signal counts. `/schedule` then logs every detected lineup
  artist with affinity + which signals caught them. Watch the dev-server stdout.

## Status (as of 2026-07-19) — LIVE share links
Share links used to be frozen snapshots: the picks were encoded into the URL
(`?s=<bitset>`), so editing your schedule after sending a link did nothing. Now a
link is `/share/<slug>` backed by KV and **keeps showing your current plan**.
- `lib/sharelink.ts` — `share:<slug>` → `{owner, ids, updatedAt}` plus a
  `sharelink:<owner>` → slug mapping so one person keeps ONE stable link forever
  (survives clearing localStorage). 1-year TTL, refreshed on every write.
- **Lazy creation**: nothing is stored until you open the share menu. Publishing
  happens on menu-open, not on the Copy click — awaiting a fetch inside the click
  handler makes Safari drop clipboard permission.
- **Auto-sync**: once a link exists, changes to what you share (locks, breaks, the
  🔥/🔮 scope toggles) write through, debounced 600ms. Never publishes an empty
  list — that would blank a link friends already hold.
- **Two-way**: friends are stored by slug and re-resolved on every `/schedule`
  load, so their edits show up in your compare view. A revoked friend link keeps
  its last-known picks and shows `· link off` rather than silently emptying.
  Re-pasting the same link updates that friend instead of duplicating the row.
- **Graceful degradation**: if KV is down, publish returns 503 and the client
  falls back to the stateless `?s=` snapshot rather than handing out a URL that
  404s. Legacy `?s=` and `?sets=` links still resolve — nothing ever shared broke.

### Security invariants — DO NOT regress these
- **Owner is always derived server-side** (`resolveShareOwner`) from the signed
  session cookie or the `manual_id` device cookie. The request body can never
  name its own owner. `findOwnedSlug` re-checks the stored record's owner before
  any write, and `revokeShare` before any delete.
- Revoke returns the **same 200 shape** whether or not it deleted, so it can't be
  used as a slug-existence oracle.
- Slugs are 72 bits of CSPRNG (12 base64url chars) and `mintUnusedSlug` re-rolls
  on collision — a collision would hand one person's link to another.
- **Two rate limits on save, deliberately**: 120/10min for *updates*, but only
  8/10min for *creating* a link (`sharenew`). A cookieless caller gets a fresh
  identity — and so a fresh year-long KV record — on every request, so without
  the second limit one host could park thousands of records a day.
- Set ids are validated against the **real lineup**, not just `^set-\d+$`. The
  share page renders only from lineup data, so share content can't carry XSS.
- Records hold no PII (ids + timestamp only) and `/api/share/resolve` never
  echoes `owner` back.
- Both share routes are `noindex, nofollow` (the old snapshot route was **not**
  before this change) and the live route is `no-store`.
- Sign-in adopts an anonymous device link into the account (same slug, ownership
  transferred) so a link already handed out survives signing in.
- Known + accepted: `clientIp()` trusts `x-forwarded-for`, so rate limits are
  spoofable by rotating that header — pre-existing across every route here, and
  nothing in this design leans on rate limiting for *security* (72-bit slugs make
  enumeration hopeless); it's a cost ceiling. Two tabs racing can mint two links
  for one owner; the older one keeps working but stops being auto-updated.
